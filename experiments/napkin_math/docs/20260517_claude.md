structure *"production-candidate format; freeze the structure"*, and

## TL;DR — what shipped between 0516 and 0517

Two PRs, one merged and one in flight.

**PR #708 — `feat/napkin-math-quick-wins` (merged):**
Five analysis blocks added to `run_monte_carlo.py`. Each block answers
a question the raw Monte Carlo tables don't:

| Block | Question answered | How |
|---|---|---|
| `binding_gate_analysis` | When a `min()` aggregate fails, which underlying gate was the binding constraint? | Per failed run, identify the dependency whose surplus is the smallest; aggregate frequency. |
| `quartile_analysis` | What is the lift in pass-probability from moving an input from its bottom quartile to its top quartile? | P(pass) conditioned on the input's quartile, top-bottom delta. |
| `required_input_thresholds` | What single-input restriction would lift a failing gate's pass rate to 80%? | Sweep input percentile cutoffs, find the tightest that satisfies the conditional pass probability ≥ 0.8. Empty list when no single-input fix is enough. |
| `missing_value_priority` | Which missing inputs are worth pinning down with real data, in priority order? | Score = `\|Δ-pp on worst gate\| × (1 − that gate's pass rate) × bound-width-ratio`. |
| `model_confidence` | How much should we trust each output? | HIGH/MEDIUM/LOW grade per output from the share of `source: data` input bounds and the average bound-width-to-base ratio. |

These five blocks become the structural data behind the new
`assessment.md`: Decision implications uses `quartile_analysis`,
Failure drivers uses `quartile_analysis` + `required_input_thresholds`,
Missing inputs uses `missing_value_priority`, Per-output confidence
uses `model_confidence`, the aggregation footnote uses
`binding_gate_analysis`. The summarizer kept the old shape in #708
(still emitted the project-manager report); the structural shift came
in #710.

PR #708 also tightened a few reporting details:
- `np.ptp` replaces `np.std == 0` for constant-array detection
  (FP-precision bug).
- `min()` aggregate magnitudes are demoted after individual gates
  in verdict tables — the verdict is meaningful, the raw number is
  not.

## Outstanding issues, in rough order of impact

### Highest leverage

1. **Compressed-vs-full-HTML head-to-head.** Unchanged from 0516. Most
   useful comparison: Nuuk Clay Workshop, where the digest extractor
   has now produced seven reviewed iterations (v27 → v33). Run the
   full-HTML extractor on the same source and diff the resulting
   `parameters.json` files.

2. **Lift `run-scenarios` into a Python runner.** The strict schema
   makes this a straight port. The current 13-reference-plan set has
   all needed `scenarios.json` files, but they were produced by an
   LLM stage that should now be deterministic.

3. **Format soak test across many different report types.** The v38
   reviewer recommended "test the format across many different
   report types" rather than further structural iteration. Worth
   running the pipeline on a deliberately heterogeneous batch
   (different currencies, mixed-unit gates, public-benefit +
   commercial + hobby/renovation domains) and looking for output
   sections that break or under-cover.

### Architectural

4. **Validator/repair loop.** The 16 structural checks in
   `validation.json` are run as fail-or-pass; on failure the
   upstream stage has to be re-run by hand. A small repair loop
   that fed structured violations back to the extractor would let
   us keep the strict schema without the manual re-prompt cycle.

5. **No regression test on the frozen v6 manifest.** The smoke test
   asserts a handful of section headings exist plus the schema
   version field; it does not check the full manifest's field set
   (including the v6 additions `known_unmodelled_existential_gates`
   and `assessment_scope_warning`) or the `schema_notes` enums.
   Worth adding a regression test on the synthetic fixture's
   `assessment.md` that locks the manifest's top-level keys + the
   `schema_notes` enums + the section list, so any unintended drift
   in `summarize_assessment.py` fails CI.

### Minor

6. **Model-collapse alert doesn't attribute cause.** Unchanged from
   0516. The Monte Carlo runner has the per-run pool state at the
   moment of the failure; could log which arguments were missing.

7. **`output/` tree gitignored.** Unchanged from 0516. Each format
   iteration regenerates 13 plans locally; none of them are
   reproducible without rerunning the upstream LLM stages. If the
   experiment is to be replicable by a new contributor, at least
   one reference plan's full output (parameters.json, bounds.json,
   scenarios.json, montecarlo.json, validation.json, assessment.md)
   should be committed as a regression baseline.

8. **The schema-v6 `basis_enum` is wider than what the pipeline
   emits.** Documented as reserved for future provenance types
   (`report_explicit`, `report_inferred`, `external_reference`,
   `manual_override`, `unknown`); none of these are currently
   produced. If they stay unused for many iterations the
   schema-notes reservation will need to be honoured or removed.

## Process insight — fix prompts, not outputs

The biggest lesson from the v40-v43 sequence is also the lesson I had
saved to memory at `feedback_fix_prompts_not_outputs.md` and then
violated:

> When a generated artifact has a flaw, fix the prompt/code that
> produced it; don't hand-patch the output — next run will reproduce
> the bug.

For v40, v41, v42 I:

1. Updated the extractor system prompt to instruct emission of
   `unmodelled_gates`.
2. Then hand-edited `parameters.json` directly with the Edit tool to
   add the unmodelled-gate entries, instead of re-running the
   extractor against the updated prompt.

This pretended the prompt update worked when in fact it had never
been tested end-to-end. Three real bugs were sitting under the
hand-patches until I finally did the clean-room v43 run:

1. **Validator regex bug.** `threshold_friendly_naming` used
   `endswith("_shortfall")` but real output names append the unit
   suffix (`_shortfall_usd`, `_gap_eur`). The check never fired on
   any real plan. Fixed to a regex matching the bad word as a
   token followed by an optional trailing unit segment:
   `r"_(gap|deficit|shortfall)(_[a-z0-9_]+)?$"`.

2. **LLM prompt-compliance gap (threshold-friendly naming).** The
   v43 agent emitted `phase2_funding_shortfall_usd` despite the
   system prompt's explicit threshold-friendly-naming rule. With
   the validator regex fixed, the next validator pass caught it
   with a WARN; a follow-up agent flipped the formula sign and
   renamed to `phase2_funding_surplus_usd`.

3. **LLM prompt-compliance gap (dead-end variables).** The v43
   agent's first pass had two unconnected missing-value entries
   despite the system prompt's "No dead-end variables" rule. The
   validator's `no_dead_end_variables` check caught them; a
   follow-up agent wired them into calculations.

None of these would have surfaced if I had kept hand-patching v44,
v45, v46. The hand-patched output was a Potemkin pipeline.

**Process rules going forward:**

1. After updating an LLM stage's system prompt, the next pipeline
   run on a real plan **must** use a freshly delegated agent
   against the updated prompt — not a copy-forward of the prior
   iteration's `parameters.json` with manual edits.

2. The validator runs **before** accepting the agent's output. If
   it reports violations, the fix is either (a) re-prompt the agent
   with the validator's report so the LLM repairs its own output,
   or (b) fix a real bug in the prompt, validator, or script. Never
   hand-edit the output to silence the validator.

3. When the same `parameters.json` is needed across multiple
   iterations for review purposes (e.g. ChatGPT format feedback on
   the `assessment.md`), copy `parameters.json` forward unchanged
   and re-run only the deterministic Python stages. The moment a
   `parameters.json` edit becomes desirable, that's the signal to
   re-run the extractor.

4. The smoke fixture under `tests/fixtures/smoke/` exercises the
   validator end-to-end; treat it as the canary for "validator
   changes don't break clean inputs".

5. ChatGPT format feedback on `assessment.md` is structural review,
   not LLM-prompt review. Substantive findings (the model is too
   narrow, a section is missing, the manifest field is misnamed)
   route to either a prompt change or a script change. They never
   route to a one-off output edit. If the only way to demonstrate
   the fix is to edit the output by hand, the fix isn't done.
